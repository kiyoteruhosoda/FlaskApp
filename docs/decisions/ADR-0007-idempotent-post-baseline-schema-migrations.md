# ADR-0007: ベースライン以降のスキーママイグレーションは冪等にする

- ステータス: Accepted
- 日付: 2026-07-19

## コンテキスト

`scripts/run_db_migrations.py` は Alembic 管理外のレガシーDB（現行モデルから
スキーマ構築済みだが `alembic_version` が無い/空）を検出すると、`init_master` へ
stamp してから `upgrade head` を実行して復旧する（自己修復。経緯は ADR-0001 と
`test_db_migrate_self_heal.py` 参照）。

この経路では、ベースライン以降の全マイグレーションが「スキーマは既に現行モデルと
一致している」DB に対して再生される。従来の後続リビジョンはデータ投入のみ
（seed・権限同期）で偶然問題にならなかったが、PR #872 で初のスキーマ変更
（`media` へのインデックス追加）を後続リビジョンとして追加した際、
`CREATE INDEX` が `already exists` で失敗し自己修復が壊れることが
`test_db_migrate_self_heal` で顕在化した。

## 決定

`init_master` より後に追加するマイグレーションは、**適用対象の変更が既に存在して
いても失敗しない（冪等である）** ことを必須とする。

- スキーマ変更: `op.create_index(..., if_not_exists=True)` /
  `op.drop_index(..., if_exists=True)` のように存在チェック付きで書く。
  `if_not_exists` が使えない操作は inspector で存在確認してからスキップする。
- データ投入: 差分投入（不足分のみ追加、既存は削除しない）とする
  （`0900277b3348_sync_role_permissions_with_master_data` と同方式）。

## 選択肢と理由

- 案A（採用）: 各マイグレーションを冪等に書く。自己修復経路・通常経路の両方で
  同一ファイルがそのまま動き、追加の分岐が要らない。`test_db_migrate_self_heal`
  が回帰を検出できる。
- 案B: 自己修復時に stamp 先を `head` にする（後続を再生しない）。レガシーDBが
  「現行モデルと完全一致」である保証はスキーマのみで、seed 等のデータ投入が
  スキップされ管理者ログイン不能などの実害が出るため不採用。
- 案C: 自己修復時のみ後続スキーマ変更を検出してスキップする仕組みを作る。
  マイグレーション本体と復旧スクリプトの二重管理になり、判定ロジック自体が
  新たな障害点になるため不採用。

## 影響

- 良い点: レガシーDB復旧・通常アップグレード・空DB新規構築のすべてが同一の
  マイグレーション列で動く。
- トレードオフ: マイグレーション作成時に冪等性を意識する必要がある。
  `test_db_migrate_self_heal` が「モデルから構築済みのDBへの再生」を検証する
  ため、冪等でないマイグレーションは CI で検出される。
- 既存への影響: なし（既存リビジョンはすべて冪等）。初の適用例は
  `b7d41c2f9a10_add_media_duplicate_lookup_indexes`。
