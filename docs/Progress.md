## 詳細

- **T14 Profile にタイムゾーン追加＋UI 現地時刻表示** — 現状、日時ずれは
  API 側の場当たり修正（naive datetime に `Z` を付与）で個別に潰しているが、
  恒久対応としてユーザーごとのタイムゾーンを Profile に持たせ、画面表示を
  その現地時刻に統一する。
  - **方針**: バックエンド（API/DB）は常に UTC で保持・送出（ISO8601 + `Z`）。
    変換は表示層（フロントエンド）でのみ行う。ドメイン/永続化を UTC に統一する
    原則は維持する。
  - **Profile**: ユーザー設定にタイムゾーン（IANA 名、例 `Asia/Tokyo`）を追加。
    - 設定キー追加は3ファイル更新ルールに従う場合はシステム設定だが、これは
      ユーザー個人設定（user_preferences 系）として持たせる想定。既存の
      プロフィール/ユーザー設定の保存経路に合わせる。未指定時のフォールバックは
      ブラウザのタイムゾーン（`Intl.DateTimeFormat().resolvedOptions().timeZone`）。
  - **フロントエンド**: 日時整形を現地時刻へ統一する。`frontend/src/utils/format.ts`
    等の整形関数に選択タイムゾーンを適用し、絶対時刻・相対時刻の両方を現地時刻
    基準で表示する。
  - **除外（重要）**: System Logs（`appdb.log` / `worker_log` 由来の表示）は
    サーバー時刻＝UTC のまま表示する。ログは時系列突合・監査用途のため現地変換
    しない。`SystemLogsPage` 等ログ系画面は本タスクの対象外。
  - **前提整備**: API が返す日時が漏れなく UTC（`Z` 付き）であること。現状
    `presentation/fastapi/` 配下に naive datetime をそのまま
    `.isoformat().replace("+00:00","Z")` している箇所が点在するため、UTC 付与を
    共通ヘルパーへ集約して横断的に統一する（本タスクの下準備）。
  - **未決定事項**: タイムゾーンの選択 UI（一覧 or 自動検出のみ）、相対時刻表示の
    しきい値、日付境界（「今日/昨日」等）の判定を現地基準にするか。着手前に確定。

- **T9 ユーザースイッチ** — 運用管理者ロールが他ユーザーに成り代わって画面を確認できる
  機能（impersonation）。監査ログ（誰がいつ誰に切り替えたか）と成り代わり中の表示、
  元ユーザーへ戻る導線が必須。認可・セッション設計に影響するため ADR を書いてから着手。
  ※ `impersonation_audit_log` テーブルと `admin:impersonate` 権限コードは T11 で追加済み。

- **T12 初期管理者フルスタックE2E** — 既存の `frontend/e2e/*.spec.ts`（Playwright）は
  全てAPIをモックし実バックエンドを起動しない方針。これとは別に、実FastAPIサーバー
  ＋実DB（SQLite、`scripts/run_db_migrations.py` でスキーマ・マスタデータ投入）＋
  ビルド済みSPA（FastAPI自身が `frontend/build` を静的配信するため別途 `vite preview`
  は不要）を起動し、`admin@example.com` / `admin` で実際にログインして
  `App.tsx` の全ルート（約30画面）を巡回するフルスタックE2Eを追加する。
  既存スイートとは別設定ファイル（例: `playwright.fullstack.config.ts` /
  `e2e-fullstack/`）に分離し、重いテストとして別ジョブで実行する想定。
  - パラメータ付きルート（`/albums/:id`, `/sessions/:sessionId`,
    `/wiki/page/:slug` 等）は事前に管理者トークンで最小限のフィクスチャを
    API経由で作成してから遷移する。
  - 判定基準: コンソールエラー無し、権限エラー/500表示無し、各画面の
    `data-testid` ルート要素が表示されること。
  - 未決定事項: CI実行タイミング（毎PR/毎マージ/nightly）、フィクスチャ
    作成範囲。着手前に方針確定。
